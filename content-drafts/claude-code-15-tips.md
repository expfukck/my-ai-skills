## 二、为什么你需要这些技巧？

### 常见的三个误区

**误区 1：Claude Code = ChatGPT + 代码编辑器**

很多人把 Claude Code 当成"会写代码的聊天机器人"，直接丢给它一个任务，然后期待完美的代码。

结果往往是：
- 代码能跑，但质量一般
- 重复修改多次才能用
- 不知道为什么有些任务做得好，有些做得差

**真相：** Claude Code 是一个需要"训练"的协作伙伴，你的使用方式直接决定了它的表现。

---

**误区 2：让 Claude 自动处理 Git**

很多新手看到 Claude Code 能执行命令，就让它自动 commit 和 push。

结果：
- 提交信息不符合团队规范
- 误提交了不该提交的文件
- 甚至覆盖了之前的工作代码

**真相：** Claude 不理解你的 Git 历史和团队规范，代码控制权必须在你手里。

---

**误区 3：把所有问题都丢给 Claude**

有些开发者过度依赖 Claude Code，简单的配置问题也要问它。

结果：
- 上下文被无关信息污染
- Claude 的"记忆"被浪费在琐碎问题上
- 真正复杂的任务反而做不好

**真相：** Claude Code 有上下文限制，要把它用在刀刃上。

## 三、15 条实战技巧

### 🛠️ 配置优化类（让 Claude 更懂你的项目）

#### 1. 善用 CLAUDE.md - 你的项目说明书

**问题：** 每次对话都要重新解释项目结构、技术栈、编码规范

**解决方案：**

在项目根目录创建 `CLAUDE.md`，Claude Code 会自动读取并遵循：

```markdown
# 项目配置

## 技术栈
- Next.js 14 (App Router)
- TypeScript (strict mode)
- TailwindCSS
- Prisma + PostgreSQL

## 编码规范
- 使用函数式组件，禁用 class 组件
- 优先使用 async/await，避免 Promise.then()
- 文件命名：kebab-case（my-component.tsx）
- 导出：优先 named export

## 项目结构
```
src/
  app/          # Next.js 路由
  components/   # 可复用组件
  lib/          # 工具函数
  types/        # TypeScript 类型定义
```

## 禁止事项
- ❌ 不要修改 prisma/schema.prisma
- ❌ 不要使用 any 类型
- ❌ 不要自动提交代码到 Git
```

**效果：**
- Claude 自动遵循这些规则
- 生成的代码符合项目规范
- 不需要每次重复说明

**进阶技巧：**
- 子目录可以有自己的 CLAUDE.md（覆盖根目录配置）
- 用链接引用外部文档（避免文件过大）

---

#### 2. 保持 CLAUDE.md 精简

**问题：** CLAUDE.md 写了 500 行，结果 Claude 反应变慢

**原因：** CLAUDE.md 会在每次对话中加载，内容太多会占用上下文

**最佳实践：**
- ✅ 只写核心规则和架构（< 200 行）
- ✅ 详细文档用链接引用
- ❌ 不要把整个 API 文档复制进来

**示例：**

```markdown
❌ 错误做法：
## API 文档
（粘贴 200 行 API 详细说明）

✅ 正确做法：
## API 文档
详见：docs/api-reference.md
核心约定：所有 API 返回格式统一为 { data, error }
```

---

#### 3. 分阶段提供上下文

**问题：** 一次性给 Claude 10 个文件，它反而抓不住重点

**技巧：** 按需提供上下文，分阶段引导

**示例：**

```
❌ 不好的做法：
"帮我看看这个项目的登录功能有什么问题"
（同时打开 auth.ts, user.ts, api.ts, schema.prisma...）

✅ 好的做法：
第一步："先看 auth.ts，我想了解登录流程"
第二步："现在看 user.ts 的 getUserById 函数，它和登录的关系"
第三步："结合这两个文件，帮我找出为什么登录后获取不到用户信息"
```

---

#### 4. 批量操作，一次性说清

**问题：** "帮我修改 user.ts" → 改完 → "再改 auth.ts" → 改完 → "再改 api.ts"

**问题在哪：** 多次往返，上下文碎片化

**最佳实践：**

```
✅ 一次性描述完整需求：
"我要给登录功能添加邮箱验证，涉及：
1. user.ts - 添加 emailVerified 字段
2. auth.ts - 登录时检查邮箱是否验证
3. api.ts - 添加发送验证邮件的接口
4. schema.prisma - 更新数据库模型

请帮我一次性完成这些修改"
```

**效果：** Claude 能看到全局，生成更协调的代码

---

#### 5. 使用 /clear 战略性重置

**问题：** 对话进行了 30 轮，Claude 开始"糊涂"了

**原因：** 上下文太长，早期信息被"遗忘"或混淆

**技巧：**
- 完成一个任务后，用 `/clear` 清空对话
- CLAUDE.md 的内容会保留
- 新任务从干净的上下文开始

**何时使用 /clear：**
- ✅ 完成一个功能模块，开始新模块
- ✅ 调试完 bug，开始写新功能
- ✅ Claude 的回答开始不相关时

### 🎯 使用策略类（提升协作效率）

#### 6. 先规划，再执行 - Plan Mode 是你的好朋友

**问题：** 复杂任务直接让 Claude 写代码，结果返工多次

**解决方案：** 使用 **Plan Mode**

**工作流：**

```
第一步：进入 Plan Mode
"我要实现用户权限系统，包括角色、权限、权限检查。
请先进入 Plan Mode，给我一个实现方案"

第二步：Claude 提供方案（不消耗执行资源）
- 数据库设计
- 文件结构
- 关键函数

第三步：你审核并调整方案

第四步：退出 Plan Mode，开始执行
```

**优势：**
- Plan Mode 消耗更少资源（更快、更便宜）
- 在动手前发现设计问题
- 避免返工

---

#### 7. 先问"怎么做"，再让它做

**问题：** 让 Claude 直接写代码，结果发现思路不对

**技巧：** 先让 Claude 解释思路

```
❌ 直接要代码：
"帮我写一个用户认证中间件"

✅ 先问思路：
"如何实现一个用户认证中间件？请说明：
1. 主要逻辑步骤
2. 需要处理的边缘情况
3. 潜在的安全问题"

→ 审核思路 → "好的，按这个思路写代码"
```

**效果：**
- 提前发现设计缺陷
- 你能学到 Claude 的思考过程
- 最终代码质量更高

---

#### 8. 利用 Subagents（子智能体）隔离任务

**概念：** Claude 可以生成子智能体处理独立任务

**场景：** 你在写功能 A，突然需要调试功能 B

**传统做法：**
```
当前对话：正在实现功能 A
突然：帮我看看功能 B 的 bug
→ 上下文被污染
→ 功能 A 的信息被功能 B 干扰
```

**使用 Subagent：**
```
当前对话：实现功能 A
新建 Subagent：调试功能 B（独立上下文）
→ 两个任务互不干扰
```

**如何使用：**
- Claude Code 会自动建议在合适时使用 subagent
- 或者明确要求："请用 subagent 处理这个独立任务"

---

#### 9. 要求测试 - 即使是简单的测试

**问题：** Claude 写的代码看起来对，但有隐藏 bug

**技巧：** 要求 Claude 写测试

```
✅ 在任务结尾加上：
"同时为这个函数写几个基础测试用例，至少覆盖：
- 正常情况
- 边缘情况（空值、null、undefined）
- 错误情况"
```

**效果：**
- 测试过程中 Claude 可能发现自己代码的 bug
- 你能快速验证代码正确性
- 有了基础测试，后续修改更安全

---

#### 10. 使用简单、流行的技术栈

**观察：** Claude Code 在某些技术栈上表现更好

**表现好的技术栈：**
- ✅ React / Next.js（流行，训练数据多）
- ✅ FastAPI / Flask（Python 简单框架）
- ✅ Express.js（Node.js 经典框架）
- ✅ SQLAlchemy / Prisma（ORM 标准）

**表现一般的技术栈：**
- ⚠️ 小众框架（训练数据少）
- ⚠️ 自创的抽象层（Claude 不理解）
- ⚠️ 过于新的技术（文档少）

**建议：**
- 如果可以选择，优先用流行技术栈
- 小众技术需要提供更详细的文档

### ⚠️ 避坑指南类（别犯这些错）

#### 11. 永远不要让 Claude 自动控制 Git

**为什么：**
- Claude 不理解你的 Git 历史
- 可能误删分支、强制推送、覆盖代码
- 提交信息不符合团队规范

**正确做法：**

```markdown
在 CLAUDE.md 中明确声明：
## 禁止事项
- ❌ 不要执行 git commit
- ❌ 不要执行 git push
- ❌ 不要修改 .git 相关文件

提示我运行 git 命令，但不要自动执行
```

**工作流：**
1. Claude 写完代码
2. Claude 提示："建议运行 git add ... && git commit -m '...'"
3. **你审核后手动执行**

---

#### 12. 代码审查是必须的 - 要求 Claude 自查

**问题：** Claude 写的代码可能有逻辑错误、性能问题

**技巧：** 让 Claude 审查自己的代码

```
✅ 在任务最后加上：
"请审查你刚才写的代码，检查：
- 是否有潜在的 bug
- 是否有遗漏的边缘情况
- 变量命名是否清晰
- 是否有性能问题"
```

**真实案例：**
```
任务："实现用户搜索功能"
Claude 写了代码

要求自查后，Claude 发现：
"刚才的代码没有处理搜索关键词为空的情况，
也没有对搜索结果做分页，可能导致性能问题。
建议修改为..."
```

**效果：** Claude 经常能发现自己的问题并修复

---

#### 13. 避免过度依赖 - 简单问题自己解决

**问题：** 一些简单的问题也问 Claude，浪费上下文

**哪些问题不该问 Claude：**
- ❌ 简单的语法错误（看文档更快）
- ❌ 配置问题（npm install、环境变量）
- ❌ 琐碎的代码格式调整

**把 Claude 用在刀刃上：**
- ✅ 复杂的业务逻辑
- ✅ 架构设计
- ✅ 难以定位的 bug
- ✅ 需要理解大量上下文的任务

**经验法则：**
- 如果你 Google 5 分钟能解决 → 自己解决
- 如果需要理解多个文件才能解决 → 让 Claude 帮忙

---

#### 14. 小心 Claude 的"创造性" - 它可能编造 API

**问题：** Claude 有时会"编造"不存在的函数或 API

**真实案例：**
```
用户："帮我用 Stripe 实现订阅功能"
Claude："可以使用 stripe.subscriptions.createWithTrial() ..."

问题：Stripe 没有 createWithTrial 这个方法！
Claude 把 create() 和 trial 概念混在一起了
```

**如何避免：**
- ✅ 对陌生的 API，查看官方文档验证
- ✅ 要求 Claude 给出文档链接
- ✅ 在 CLAUDE.md 中提供正确的 API 文档链接

**最佳实践：**
```
要求 Claude："使用 Stripe 官方文档中的 API，
不要假设或推测 API 名称"
```

---

#### 15. 语音输入加速提示词 - 但要注意准确性

**技巧：** 使用语音输入工具（如 SuperWhisper、macOS 听写）

**优势：**
- 快速输入长提示词
- 适合描述复杂需求
- 解放双手

**注意事项：**
- ⚠️ 检查语音识别错误（特别是技术术语）
- ⚠️ 代码片段还是手打更准确
- ⚠️ 可以语音描述需求，然后手动补充代码示例

**推荐工具：**
- macOS：系统自带听写（按两次 Fn）
- Windows：PowerToys Text Extractor
- 专业工具：SuperWhisper, Wispr Flow

## 四、最佳实践组合方案

### 🎯 方案 A：新项目启动

```markdown
第1步：创建 CLAUDE.md
- 技术栈
- 编码规范
- 项目结构

第2步：Plan Mode 规划架构
- 目录结构
- 核心模块
- 数据流

第3步：分模块实现
- 每个模块用 /clear 隔离
- 先问思路，再写代码
- 要求测试

第4步：人工审核 + Git 提交
- 审查代码
- 运行测试
- 手动 git commit
```

---

### 🎯 方案 B：调试复杂 Bug

```markdown
第1步：先问 Claude 思路
"如何定位这个 bug？需要检查哪些地方？"

第2步：分阶段提供上下文
- 先看报错相关文件
- 再看相关模块
- 逐步缩小范围

第3步：要求 Claude 解释根因
"为什么会出现这个问题？"

第4步：修复 + 测试
- 写修复代码
- 写测试用例防止回归
```

---

### 🎯 方案 C：重构老代码

```markdown
第1步：要求 Claude 分析现有代码
"分析这个文件的问题和改进点"

第2步：Plan Mode 规划重构方案
- 保持功能不变
- 改进代码结构
- 列出风险点

第3步：分步重构
- 一次改一个小模块
- 每步都运行测试
- 用 /clear 避免上下文混乱
```
